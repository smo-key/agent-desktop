# OpenSpec Interview + Spec-Sync Add-on

Two update-safe improvements to the OpenSpec workflow, in a form you can drop
into any other OpenSpec project â€” **without editing any OpenSpec source files**:

1. **Interview gate** (`config.yaml`) â€” the **propose** step runs an
   interactive, interview-style requirements gathering before it drafts
   `proposal.md`, then presents a **detailed scope summary for approval before
   writing specs**.
2. **Spec-sync discipline** (`CLAUDE.md`) â€” follow-up messages during
   implementation keep the artifacts in sync, and a definition-of-done gate
   ensures specs are current before a change is archived.

Two files because they cover different phases (see "Why two files" below).

## Why this approach

OpenSpec generates per-tool skill files (e.g. `.claude/skills/openspec-*/SKILL.md`)
and ships a packaged workflow schema. Both are **regenerated** by `openspec update`
/ `openspec init`, so editing them directly means your changes get clobbered.

Instead, OpenSpec reads an optional **`openspec/config.yaml`** on every command.
Each step skill (propose, new-change, continue, ff) runs
`openspec instructions <artifact> --json` and obeys the `rules` it returns. The
`rules` map in `config.yaml` is **additive** to the built-in schema, keyed by
artifact id â€” so:

- âś… No source files are modified.
- âś… `openspec update` leaves `config.yaml` alone.
- âś… Upstream schema improvements still flow through (rules are appended, not a fork).
- âś… It's one small file â€” trivial to share or copy between projects.

> Tested against openspec CLI `1.3.1`. Schema *fork* commands are experimental;
> this add-on deliberately avoids them and uses the stable `config.yaml` `rules`
> field instead.

## Why two files

`config.yaml` `rules`/`context` are only injected into **artifact creation**
(proposal/specs/design/tasks). They do **not** reach the `apply` phase, and the
`verify`/`archive` skills don't read `config.yaml` at all. The spec-sync
discipline must govern those phases, so it lives in `CLAUDE.md`, which is always
in the agent's context (and is likewise never regenerated by `openspec update`).

| File | Reaches | Purpose |
|------|---------|---------|
| `openspec/config.yaml` | proposal/specs/design/tasks creation | interview gate on the proposal + scope-review gate before specs |
| `CLAUDE.md` | every phase (apply, verify, archive) | keep specs in sync + close-out gate |

## Install in another project

1. Copy `config.yaml` to `<that-project>/openspec/config.yaml`.
   - If the project already has an `openspec/config.yaml`, merge the `rules:`
     block (and keep its existing `schema:` value) instead of overwriting.
2. Copy the contents of `CLAUDE.md` (the "OpenSpec change discipline" section)
   into the project's root `CLAUDE.md`, creating it if absent.
3. Commit both. Anyone who clones the repo gets the behavior automatically â€” no
   per-machine setup.
4. Verify the interview gate is wired up:
   ```bash
   openspec instructions proposal --change <any-existing-change> --json \
     | python3 -c "import sys,json;print('\n'.join(json.load(sys.stdin).get('rules') or ['<none>']))"
   ```
   You should see the interview rules listed.

## Customize

- **Change scope:** add other artifact ids under `rules:` (`specs`, `design`,
  `tasks`) to gate those steps too.
- **Apply to every artifact:** put guidance under the top-level `context:` field
  instead of `rules:` â€” it's injected into *all* artifact instructions.
- **Soften the gate:** reword the first rule from "MANDATORY" to a nudge that
  allows skipping when the request is already detailed.

## Config reference (`openspec/config.yaml`)

| Field | Purpose |
|-------|---------|
| `schema` | Which workflow schema to use (`spec-driven` is the built-in default). |
| `context` | Free-text injected into **every** artifact's instructions. |
| `rules.<artifactId>` | List of extra rule strings injected for that artifact only. Additive to the schema. |
