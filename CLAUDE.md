# Project instructions

## OpenSpec change discipline — keep specs in sync

These rules apply whenever an OpenSpec change is active (a directory exists under
`openspec/changes/<name>/`). They are deliberately kept here, not in the
generated `.claude/skills/openspec-*` files or `openspec/config.yaml`, because
they must govern the **apply**, **verify**, and **archive** phases — which
`config.yaml` rules do not reach. This file is never regenerated by
`openspec update`, so it is safe to edit and share.

### During implementation — follow-ups must update the artifacts

Treat a change's artifacts (`proposal.md`, `specs/**`, `design.md`, `tasks.md`)
as the source of truth. When a follow-up message changes **requirements or
scope** — not merely how to implement something already specified — update the
affected artifact **alongside** (ideally before) the code, in the same turn:

- New, changed, or removed **behavior** → update the delta spec(s) under
  `openspec/changes/<name>/specs/<capability>/spec.md` using
  `## ADDED` / `## MODIFIED` / `## REMOVED` / `## RENAMED Requirements`, each
  requirement with at least one `#### Scenario:` (WHEN/THEN, exactly 4 hashes).
- New work to do → add `- [ ] N.M …` checkboxes to `tasks.md` (and check them off
  as you complete them).
- Changed **why / what / scope** → update `proposal.md` (and `design.md` if a
  technical decision changed).

Never let code and specs diverge silently. If you are unsure whether a follow-up
is a scope change (artifact update needed) or just an implementation detail
(no artifact change), **ask** rather than guess. After updating artifacts, run
`openspec validate <name>` to confirm they are still well-formed.

### Closing out a change — definition of done

A change is not "done" until all of the following hold, in order:

1. **Tasks complete** — every `- [ ]` in `tasks.md` is checked.
2. **Verify** — run the `openspec-verify-change` skill, and explicitly reconcile
   **conversation drift**: any behavior discussed or implemented in this session
   that is not yet reflected in `specs/`, `tasks.md`, or `proposal.md` must be
   added now. Do not pass verification with known, unrecorded drift.
3. **Archive** — run the `openspec-archive-change` skill. This promotes the
   change's delta specs into `openspec/specs/` (the durable, up-to-date specs)
   and moves the change to the archive.

To update main specs mid-stream without archiving, use the
`openspec-sync-specs` skill. The user can trigger close-out by running
`/openspec-verify-change` then `/openspec-archive-change`, or by asking to
"wrap up" / "archive" the current change — either way, apply this gate.
